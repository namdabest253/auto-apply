# Phase 03.1: Internship Discovery Improvements - Research

**Researched:** 2026-03-09
**Domain:** Job board API scraping, internship filtering, Lever ATS integration
**Confidence:** HIGH

## Summary

This phase addresses a critical yield problem: only 47 out of ~5800 scraped jobs are internships because the Greenhouse scraper fetches ALL jobs and applies no internship-level filtering. The fix requires three interventions: (1) add positive/negative keyword filtering to the Greenhouse scraper, (2) expand the company list from 23 to 50+ companies with verified internship programs, and (3) implement a Lever API scraper following the identical adapter pattern.

The Lever public Postings API (`api.lever.co/v0/postings/{company}`) is structurally very similar to Greenhouse's public boards API -- both are unauthenticated JSON endpoints returning all published jobs for a company. The Lever API additionally supports server-side filtering by `commitment`, `team`, `department`, and `level` query parameters, though these depend on companies tagging their postings correctly. Client-side internship filtering (same as Greenhouse) is the reliable approach.

**Primary recommendation:** Implement internship filtering as a shared utility function used by both Greenhouse and Lever scrapers, expand company lists with verified slugs, and build LeverScraper as a near-copy of GreenhouseScraper with field mapping differences.

### Pattern 1: Shared Internship Filtering
**What:** Extract internship detection and exclusion logic into a reusable module
**When to use:** Any scraper that needs to filter for intern-level roles
**Example:**
```typescript
// src/lib/scrapers/filters.ts

const INTERN_POSITIVE_PATTERNS = [
  /\bintern\b/i,
  /\binternship\b/i,
  /\bco-?op\b/i,
];

const INTERN_NEGATIVE_PATTERNS = [
  /\bsenior\b/i,
  /\bstaff\b/i,
  /\bprincipal\b/i,
  /\blead\b/i,
  /\bmanager\b/i,
  /\bdirector\b/i,
  /\bphd\b/i,
  /\bmba\b/i,
  /\bvp\b/i,
  /\bhead of\b/i,
  /\bfellow\b/i,
];

export function isInternshipRole(title: string, department?: string): boolean {
  const text = `${title} ${department || ""}`;
  const hasPositive = INTERN_POSITIVE_PATTERNS.some((p) => p.test(text));
  const hasNegative = INTERN_NEGATIVE_PATTERNS.some((p) => p.test(title));
  return hasPositive && !hasNegative;
}

export function filterInternships<T extends { title: string }>(
  jobs: T[],
  getDepartment?: (job: T) => string | undefined
): T[] {
  return jobs.filter((job) =>
    isInternshipRole(job.title, getDepartment?.(job))
  );
}
```

### Pattern 2: LeverScraper Following Adapter Interface
**What:** New scraper class implementing `ScraperAdapter` for Lever's public Postings API
**When to use:** Lever company job boards at `api.lever.co/v0/postings/{company}`
**Example:**
```typescript
// src/lib/scrapers/lever.ts
import type { ScraperAdapter, SearchParams, DiscoveredJob } from "./types";
import { LEVER_COMPANIES } from "./constants";
import { randomDelay } from "./stealth";
import { isInternshipRole } from "./filters";

const LEVER_API = "https://api.lever.co/v0/postings";

export class LeverScraper implements ScraperAdapter {
  platform = "lever";

  async discover(params: SearchParams): Promise<DiscoveredJob[]> {
    const allJobs: DiscoveredJob[] = [];

    for (const company of LEVER_COMPANIES) {
      try {
        const url = `${LEVER_API}/${company.slug}?mode=json`;
        const res = await fetch(url);

        if (!res.ok) {
          console.warn(`[LeverScraper] Failed ${company.name} (${res.status})`);
          continue;
        }

        const postings = await res.json();
        const jobs: DiscoveredJob[] = (postings || []).map((p: any) => ({
          externalUrl: p.hostedUrl,
          platform: "lever" as const,
          title: p.text,
          company: company.name,
          location: p.categories?.location ?? null,
          datePosted: p.createdAt ? new Date(p.createdAt) : null,
          descriptionHtml: p.description ?? null,
          descriptionText: p.descriptionPlain ?? null,
          salary: p.salaryRange
            ? `${p.salaryRange.currency} ${p.salaryRange.min}-${p.salaryRange.max}/${p.salaryRange.interval}`
            : null,
          metadata: {
            leverId: p.id,
            commitment: p.categories?.commitment,
            team: p.categories?.team,
            department: p.categories?.department,
            workplaceType: p.workplaceType,
            leverSlug: company.slug,
          },
        }));

        // Filter for internships
        const filtered = jobs.filter((j) =>
          isInternshipRole(j.title, (j.metadata as any)?.team)
        );

        allJobs.push(...filtered);
        await randomDelay(500, 1500);
      } catch (error) {
        console.warn(`[LeverScraper] Error ${company.name}:`, (error as Error).message);
        continue;
      }
    }

    return allJobs;
  }
}
```

### Anti-Patterns to Avoid
- **Filtering only on title:** Check department/team fields too -- some companies list "Internship Program" as department but use generic titles
- **Server-side filtering on Lever:** The `commitment` and `level` params depend on companies actually setting these fields; many do not. Always filter client-side.
- **Separate filtering logic per scraper:** Duplicating intern/exclusion patterns in each scraper leads to drift. Use shared `filters.ts`.
- **No `co-op` handling:** Many companies post "Co-op" positions that are functionally internships. Include `co-op` in positive patterns.

## Common Pitfalls

### Pitfall 1: Regex Word Boundary on "intern"
**What goes wrong:** Pattern `/intern/i` matches "internal", "international", "internet"
**Why it happens:** Missing word boundary in regex
**How to avoid:** Use `/\bintern\b/i` or `/\binternship\b/i` with word boundaries
**Warning signs:** Non-internship jobs appearing with "Internal Tools Engineer" or "International Sales" titles

### Pitfall 2: Lever API Returns Array, Not Object
**What goes wrong:** Code tries `data.postings` or `data.jobs` and gets undefined
**Why it happens:** Unlike Greenhouse (`{ jobs: [...] }`), Lever returns a raw array `[...]`
**How to avoid:** `const postings = await res.json()` -- no property access needed
**Warning signs:** Empty results despite valid company slug

### Pitfall 3: Lever `createdAt` Is Epoch Milliseconds
**What goes wrong:** Date parsing produces wrong dates
**Why it happens:** Lever uses Unix epoch milliseconds (e.g., `1753220471930`), not ISO 8601 strings
**How to avoid:** `new Date(posting.createdAt)` works directly since JS Date constructor accepts ms
**Warning signs:** Dates showing as 1970 or NaN

### Pitfall 4: Greenhouse API Uses Two Different Base URLs
**What goes wrong:** Some companies return 404 on `boards-api.greenhouse.io`
**Why it happens:** Greenhouse has two hosting patterns: `boards-api.greenhouse.io` and `job-boards.greenhouse.io`
**How to avoid:** The current API URL (`boards-api.greenhouse.io/v1/boards/{slug}/jobs`) works for all companies tested. The `job-boards.greenhouse.io` pattern is for the HTML career page, not the API.
**Warning signs:** 404s from specific companies

### Pitfall 5: Over-Aggressive Negative Filtering
**What goes wrong:** "Senior Intern" or "Lead Intern" titles get excluded
**Why it happens:** Negative pattern matches without considering the intern context
**How to avoid:** Apply negative patterns only to the portion of the title excluding "intern" keyword, OR accept that these edge cases are rare enough to not matter
**Warning signs:** Missing legitimate intern roles

### Pitfall 6: Lever 429 Rate Limiting
**What goes wrong:** API returns 429 Too Many Requests
**Why it happens:** Lever enforces rate limits (documented as 2 req/sec for POST, undocumented for GET)
**How to avoid:** Use `randomDelay(500, 1500)` between company requests (same as Greenhouse pattern)
**Warning signs:** Sudden empty responses or HTTP 429 status codes

## Lever API Reference

**Verified via live API testing (2026-03-09).**

### Endpoints
| Method | URL | Auth | Response |
|--------|-----|------|----------|
| GET | `https://api.lever.co/v0/postings/{company}?mode=json` | None | JSON array of posting objects |
| GET | `https://api.lever.co/v0/postings/{company}/{id}` | None | Single posting object |

### Key Response Fields (per posting)
| Field | Type | Maps To (DiscoveredJob) |
|-------|------|------------------------|
| `id` | string (UUID) | `metadata.leverId` |
| `text` | string | `title` |
| `hostedUrl` | string | `externalUrl` |
| `categories.location` | string | `location` |
| `categories.commitment` | string | `metadata.commitment` |
| `categories.team` | string | `metadata.team` |
| `categories.department` | string | `metadata.department` |
| `createdAt` | number (epoch ms) | `datePosted` (via `new Date()`) |
| `description` | string (HTML) | `descriptionHtml` |
| `descriptionPlain` | string | `descriptionText` |
| `salaryRange` | object `{currency, interval, min, max}` | `salary` |
| `workplaceType` | string | `metadata.workplaceType` |

### Query Parameters
| Param | Purpose | Note |
|-------|---------|------|
| `mode=json` | Force JSON response | Required (default is HTML) |
| `limit` | Max results per request | Optional, for pagination |
| `skip` | Offset for pagination | Optional |
| `commitment` | Filter by type (e.g., "Intern") | Unreliable -- companies may not set this |

### Differences from Greenhouse API
| Aspect | Greenhouse | Lever |
|--------|-----------|-------|
| Base URL | `boards-api.greenhouse.io/v1/boards/{slug}/jobs` | `api.lever.co/v0/postings/{slug}` |
| Response shape | `{ jobs: [...] }` | `[...]` (raw array) |
| Title field | `job.title` | `posting.text` |
| URL field | `job.absolute_url` | `posting.hostedUrl` |
| Date field | `job.updated_at` (ISO string) | `posting.createdAt` (epoch ms) |
| Description | `job.content` (HTML only) | `posting.description` (HTML) + `posting.descriptionPlain` |
| Location | `job.location.name` | `posting.categories.location` |
| Content param | `?content=true` needed for description | Description always included |
| Departments | `job.departments[]` array of objects | `posting.categories.team` string |
| Salary | Not provided | `posting.salaryRange` object |

## Open Questions

1. **Pagination for large Lever boards**
   - What we know: Lever API supports `skip` and `limit` params for pagination
   - What's unclear: Whether any company has so many postings that we need to paginate
   - Recommendation: Start without pagination; add if a company returns 0 results due to default limit. Most companies have <500 total postings, and Lever appears to return all by default.

2. **EU-hosted Lever companies**
   - What we know: Lever has a separate EU endpoint at `api.eu.lever.co`
   - What's unclear: Which companies use the EU endpoint
   - Recommendation: Start with `api.lever.co` only. If a known company returns 404, try the EU endpoint as fallback. This can be added later if needed.

3. **Greenhouse `job-boards.greenhouse.io` vs `boards-api.greenhouse.io`**
   - What we know: Some job URLs use `job-boards.greenhouse.io` (newer pattern)
   - What's unclear: Whether the API endpoint ever differs
   - Recommendation: The API at `boards-api.greenhouse.io` works for all verified companies. No change needed.
