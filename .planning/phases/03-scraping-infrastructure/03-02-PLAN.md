<task type="auto">
  <name>Task 1: BullMQ queue + worker process + Docker service + worker tests</name>
  <files>src/workers/queue.ts, src/workers/scrape.worker.ts, src/workers/scrape.worker.test.ts, docker-compose.yml, package.json</files>
  <action>
1. Install BullMQ and ioredis: `npm install bullmq ioredis`

2. Create src/workers/queue.ts:
   - Create IORedis connection factory function `createRedisConnection()` using `process.env.REDIS_URL` with `maxRetriesPerRequest: null` (CRITICAL for BullMQ -- see pitfall in RESEARCH.md).
   - Create and export `scrapeQueue = new Queue('scrape', { connection })`.
   - Export the connection factory so worker can create its own connection.

3. Create src/workers/scrape.worker.ts:
   - Import both scrapers (IndeedScraper, GreenhouseScraper), dedup helper (filterNewJobs), and Prisma client.
   - Create Worker on 'scrape' queue with concurrency: 1 (browser resource constraint).
   - Job data shape: `{ userId: string, runId: string, searchParams: SearchParams }`.
   - Worker processor:
     a. Update ScrapeRun status to "running"
     b. Run both scrapers with the searchParams. Run them sequentially (Indeed first, then Greenhouse) since Indeed uses browser resources. Wrap each in try/catch -- if one fails, continue with the other (partial-safe saving per user decision).
     c. Use filterNewJobs to deduplicate discovered jobs against existing DB entries. For each new DiscoveredJob, create in JobListing table with `prisma.jobListing.create` including userId and scrapeRunId.
     d. Update ScrapeRun: status "completed", jobsFound count (new jobs only), completedAt, duration.
     e. On error: update ScrapeRun status to "failed", store error in errors JSON field.
   - Configure custom backoff strategy per user decision: 30000ms, 60000ms, 300000ms for attempts 1-3.
   - Add worker event handlers: 'completed', 'failed' for logging.

4. Create src/workers/scrape.worker.test.ts:
   - Mock BullMQ Worker to capture the processor function.
   - Mock both scrapers (IndeedScraper, GreenhouseScraper) to return fake DiscoveredJob arrays.
   - Mock filterNewJobs to return the input (all new).
   - Mock Prisma client (prisma.scrapeRun.update, prisma.jobListing.create).
   - Test: processor updates ScrapeRun to "running" at start.
   - Test: processor calls both scrapers and saves results.
   - Test: processor updates ScrapeRun to "completed" with correct jobsFound count.
   - Test: when one scraper fails, the other still runs and results are saved (partial-safe).
   - Test: on total failure, ScrapeRun status set to "failed" with error.

5. Add to package.json scripts:
   - `"worker:scrape": "tsx src/workers/scrape.worker.ts"`
   - `"worker:scrape:watch": "tsx --watch src/workers/scrape.worker.ts"`

6. Update docker-compose.yml:
   - Add a `worker` service using the same build context and Dockerfile as `app`.
   - Command: `npx tsx src/workers/scrape.worker.ts`
   - Same environment variables (DATABASE_URL, REDIS_URL).
   - depends_on: postgres (healthy), redis (healthy).
   - No port mapping needed (worker doesn't serve HTTP).
  </action>
